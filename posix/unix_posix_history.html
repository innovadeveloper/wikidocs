Historia de Unix y POSIX

De la Fragmentación al Estándar Universal

¿Qué es Unix?

Unix no es un solo sistema operativo, sino una familia de sistemas operativos que comparten una filosofía de diseño y una API común en lo esencial.
Nació en los laboratorios Bell Labs (AT&T) en los años 70
Diseñado para ser multitarea (varios procesos a la vez)
Diseñado para ser multiusuario (varias personas usando el sistema)
De Unix derivaron Linux, macOS, BSD, AIX, HP-UX, Solaris, Android

🔹 ¿Por qué "cada Unix tenía APIs diferentes"?

Aunque todos eran "Unix" o "Unix-like", cada fabricante (IBM, HP, Sun, Apple, etc.) fue creando su propia versión de Unix.

Cada uno tenía el mismo núcleo de ideas (procesos, archivos, permisos, etc.), pero exponían sus propias APIs para operaciones básicas como:

Manejo de procesos
fork, exec, wait
Manejo de redes
socket, bind, listen
Señales
kill, signal
Entrada/Salida
read, write

Resultado: Un programa escrito para Solaris podía no compilar en AIX, aunque ambos fueran "Unix".

System V (AT&T)
#include <fcntl.h> int fd = open("/etc/passwd", O_RDONLY); if (fd == -1) { perror("open failed"); }
BSD Unix (Berkeley)
#include <sys/file.h> int fd = open("/etc/passwd", O_RDONLY, 0); if (fd < 0) { fprintf(stderr, "Cannot open file\n"); }
XENIX (Microsoft)
#include <xenix.h> int fd = xopen("/etc/passwd", X_READ); if (fd == XERR) { xerror("File open failed"); }

Problemas:
  • Código lleno de #ifdefs
  • Testing en múltiples plataformas
  • Mantenimiento complicado
  • Bugs específicos por plataforma

🔹 La solución: POSIX

Para resolver el caos, en los 80s se creó POSIX (Portable Operating System Interface), un estándar que definía una API común para sistemas tipo Unix.
Código Portable El mismo código en C compilable en Linux, macOS, BSD, etc.
APIs Estándar pthread_create, fork, open, read, write son parte de POSIX
Reducción de Costos Menos tiempo de desarrollo y testing
Mantenimiento Simple Un solo código base para múltiples plataformas
